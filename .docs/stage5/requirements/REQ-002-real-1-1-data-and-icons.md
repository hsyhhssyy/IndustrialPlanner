## 背景与目标

Stage5 需要将现有过渡性内容切换为真实游戏 1.1 版本口径，确保数据、图标与用户可见行为一致；同时将超时空相关内容从“当前迁移主线”中收口，但保留未来仍可能继续使用的开关能力。

## 当前口径

1. 真实 1.1 配方为唯一目标基线。
2. 正式图标应覆盖设备、物品、配方、蓝图等关键展示场景。
3. 超时空相关内容不再作为 Stage5 的迁移主线，但当前不要求移除现有开关入口。
4. 数据切换完成后，仍需治理与真实 1.1 迁移直接相关的过渡逻辑，避免长期双轨维护。
5. 对历史持久化数据中的未知物品，不做兼容映射；一律回退为“未配置物品”。
6. 对历史存档、持久化蓝图、导入蓝图中的未知设备，不做旧 ID 迁移；一律忽略并删除。
7. 设备资源与物品资源必须分开处理：原始设备图标放在 `public/original/device-icons/` 并导出到 `public/device-icons/*.webp`，原始设备精灵放在 `public/original/sprites/` 并导出到 `public/sprites/*.webp`，原始物品图标放在 `public/original/itemicon/` 并导出到 `public/itemicon/*.webp`；不得为设备补写 `itemicon`。
8. 物品选择器允许增加辅助过滤开关以降低噪音，但默认行为必须保守；当前已确认“隐藏瓶装液体”开关默认关闭，并放在“全部物品”分组标题右侧。
9. 设备精灵如果存在低透明度外边缘，正式入库前应按 alpha 阈值裁切，不能只依赖默认 `trim()`。
10. 行为复用型正式设备若采用独立 ID，必须同步更新分类、配置面板逻辑与运行时逻辑，不能只换图。

## 实施拆解

1. 建立真实 1.1 配方数据源与当前数据的差异清单。
2. 建立正式图标资源与现有引用点的映射清单。
3. 盘点与真实 1.1 迁移直接相关的超时空遗留内容：
   - 用户入口；
   - 门禁开关；
   - 过渡数据分支；
   - UI 标签与提示文案；
   - 依赖其存在的兼容逻辑。
4. 迁移物品时，优先采用“新增正式物品 → 替换引用 → 删除旧物品 ID → 清理持久化未知物品”的单向流程，不保留长期别名兼容。
5. 迁移设备时，若旧设备 ID 已不再受支持，则直接依赖“未知设备删除”规则处理历史数据，不新增旧 ID 到新 ID 的兼容映射。
6. 处理资源时，严格区分：
  - 原始物品：`public/original/itemicon/<itemId>.*`
  - 原始设备图标：`public/original/device-icons/<deviceTypeId>.*`
  - 原始设备精灵：`public/original/sprites/<deviceTypeId>.*`
  - 运行时物品：`public/itemicon/<itemId>.webp`
  - 运行时设备图标：`public/device-icons/<deviceTypeId>.webp`
  - 运行时设备精灵：`public/sprites/<deviceTypeId>.webp`
7. 对物品选择器等高噪音入口，可增加默认关闭的临时过滤开关；当前已登记“隐藏瓶装液体”开关。
8. 若设备精灵源图存在低透明度边缘，使用 alpha 阈值裁切生成正式 `sprite`，避免默认裁切失效。
9. 若新增正式设备 ID 复用旧设备行为，必须同步扩展配置面板、筛选器状态与运行时初始化逻辑。
10. 完成替换后，逐项回归关键界面与仿真链路。

## 验收标准

1. 配方输入、输出、耗时、能耗、适用设备与目标 1.1 版本一致。
2. 正式图标在关键界面展示正确，无明显缺图或错图。
3. 真实 1.1 数据、正式图标、正式设备分类与运行时逻辑完成收口。
4. 不再依赖旧 `chrono` 物品兼容映射支撑主流程。
5. 物品选择器“全部物品”分组标题右侧存在“隐藏瓶装液体”开关，默认关闭；开启后物品列表中所有瓶装液体被隐藏，且布局无错位或遮挡。
6. 未知物品配置不会继续保留；读取到未知物品时应回退为未配置状态。
7. 未知设备不会被自动映射到新 ID；历史布局、持久化蓝图与导入蓝图读到未知设备时应直接删除。
8. 设备资源替换不会误写到 `itemicon` 路径。
9. 正式设备若有独立 ID，则分类、图标、精灵、配置面板与运行时逻辑保持一致。
10. 正式设备精灵看起来不应残留大面积透明边框。
