# S5-RQ-003 可维护性清理与模块拆分

## 文档内容范围、结构与格式约定（必读）

### 1. 本文档负责什么

本文档用于沉淀 S5-RQ-003 上的全部结构化记忆，并采用“状态视图 + 时间线附录”的混合写法，只负责本需求相关的：

1. 高复杂度模块识别。
2. 面向业务价值的拆分、清理与去重复。
3. 与重构相关的风险、边界、验收与回归记录。

本文档不负责：

1. 为重构而重构的纯形式目录调整。
2. 记录无业务目标支撑的大规模改名。
3. 替代其他需求记录其实现细节。

### 2. 本文档固定结构

1. 文档内容范围、结构与格式约定
2. 需求卡片
3. 背景与目标
4. 当前口径
5. 当前状态视图
6. 候选清理项
7. 验收标准
8. 测试记录
9. bug / 修复记录
10. 中断与恢复记录
11. 需求拉扯与决策时间线
12. 待确认问题
13. 状态更新

### 3. 本文档记录格式

1. 文档前半部分优先给出当前最值得做的清理项，而不是完整历史回放。
2. 每个清理项必须写明业务收益，不接受“看起来更整洁”作为唯一理由。
3. 每个清理项必须写明改动边界与回归面。
4. 若某项清理服务于其他需求，需注明关联需求编号。
5. 若重构过程中触发跨需求问题，同步登记到 [../bugs.md](../bugs.md)。

## 背景与目标

项目已经积累了一批历史实现、过渡数据与结构复杂度。Stage5 需要做一批高价值清理，但前提是服务主线交付，而不是脱离业务的全面重写。

## 当前口径

1. 重构目标是降低维护成本，而不是追求形式上的完美结构。
2. 只处理高价值、高耦合、高重复、与 Stage5 主线直接相关的区域。
3. 清理后必须保证主流程行为不变，且无 P0 / P1 回归。
4. 界面完全重构不并入本需求，独立由 `S5-RQ-009` 承载；本需求仅处理支撑主线与支撑界面改造所需的结构清理。
