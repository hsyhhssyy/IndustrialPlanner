## 背景与目标

当前项目已经积累了多轮功能迭代，界面在布局层级、视觉语言、面板结构、弹窗样式与反馈一致性上存在明显历史痕迹。用户希望将“界面完全重构”升级为 Stage5 的独立需求，以便在不改变业务目标的前提下，系统性重做界面体验，而不是零散修补。

当前新目标进一步明确为：整体界面框架参考 VSCode 的工作台布局思路，形成“顶栏 + 左侧工具条 + 主工作区 + 右侧面板 + 状态栏”的稳定骨架；当前不要求补齐 VSCode 式底部面板。

## 当前口径

1. 本需求是“界面层完全重构”，不是领域逻辑或仿真引擎的全面重写。
2. 重构目标是统一信息架构、视觉样式、交互反馈与组件边界，提升可用性、一致性与后续迭代效率。
3. 核心业务主流程必须保持可用；若某处交互行为需要改变，必须在本需求内明确记录并标注影响面。
4. 当前布局目标参考 VSCode：保留顶部、左侧、右侧、状态栏四段式结构，暂不引入底部面板。
5. 废弃当前左右侧抽屉的激活逻辑，改为在顶栏提供左右面板显式 toggle 入口。
6. 顶栏中的左右面板控制样式参考 VSCode 标题栏中的面板切换按钮：使用并列的小型 toggle button，强调开关态，而不是大尺寸普通按钮。
7. 左侧改为类似 VSCode Activity Bar 的不可关闭工具条，首批固定承载 3 个工具功能：`放置`、`删除`、`蓝图`；与 VSCode 不同的是，每个工具按钮需在图标下方显示中文文字标签。
8. 左侧这 3 个工具按钮不再只是“模式开关”，而是像 VSCode 左栏一样承担 View 切换职责：点击不同按钮后，切换并展示该模式对应的子 Panel 内容。
9. 左侧 View 与现有业务模式保持一一对应，优先复用当前模式切换后的左侧 Panel 内容与事件链路；如无必要，不在首轮重构中改写底层业务语义，仍以“模式切换”为本质。
10. 右侧改为支持折叠分组的面板容器，首批至少支持在同一容器内折叠/展开“基地选择”和“当前选中”两个区域，交互体验参考 VSCode 在 View Container 内的折叠 section。
11. 原百科内容按职责拆分为两个独立入口：`工具箱` 与 `帮助`，不再合并为同一个入口。
12. `工具箱` 继续使用 `dialog` 呈现，承载偏资料与工具的内容；首批固定包含 3 个 tab：`设备一览`、`物品一览`、`产线规划器`。
13. `帮助` 继续使用 `dialog` 呈现，承载偏说明与教学的内容；首批固定包含 3 个 tab：`操作指南`、`进阶说明`、`按键说明`。
14. 当前界面中的长段操作提示文案——“编辑模式：拖拽移动设备，按住框选，按R旋转，按Ctrl+C复制，按住中键移动视角。”——不再直接悬挂在主界面显著区域，应迁入 `帮助` dialog 中的“按键说明” tab。
15. 左侧工具条底部功能区新增 `工具箱`、`帮助`、`设置` 三个入口，作为与顶部“放置 / 删除 / 蓝图”主工具 View 分离的底部功能区。
16. `工具箱` 与 `帮助` 两个入口都从原顶部栏位置迁移到左侧工具条底部，并保持 dialog 打开形态。
17. 左侧 `放置 / 删除 / 蓝图` 三个 View 可进一步细化为更稳定的 section 结构，以提升信息分层、可扫描性与扩展性。
18. 左侧工具条底部新增一个类似 VSCode 的“设置”入口，用于承载少量全局设置项；当前已包含“语言”“超时空模式”开关、“仿真速度限制”和 `Ayu Dark / Ayu Light` 主题切换。
19. 设置入口打开后使用 `dialog` 呈现，而不是侧边小面板、弹出菜单或独立设置视图。
20. 设置数据应从零散 localStorage key 逐步收敛到单一 `settings` JSON 对象中存储；首版允许通过兼容迁移读取旧 key，再统一写回 `settings`。
21. 当前实现以实际交付为准：设置 UI 已落地，且已支持 `Ayu Dark` / `Ayu Light` 真实切换；默认主题基线仍为 `Ayu Dark`。
22. 设置页当前新增一个面向性能调优的滑条型设置“仿真速度限制”，用于控制“每帧最大 tick 数”；其取值范围固定为 `4-24`，默认值为 `8`，并通过提示文案同时说明“电脑性能越好可以适当调高”以及“FPS 低于 `30` 时仿真调试里的仿真秒数可能不够准确”。
23. 界面主题配色可参考 VSCode 的 `Ayu Dark` / `Ayu Light` 风格，优先借鉴其整体层级、明度关系与强调色策略；当前允许主题同步调整画布中的辅助视觉元素（如网格与放置预览），但不修改设备、物流和正式世界渲染的业务配色口径。
24. 本轮主题改造范围覆盖工作台 UI 外壳、面板、按钮、表单、浮层、列表、状态栏，以及画布中的辅助视觉元素（如网格与放置预览）；不触及设备着色、覆盖层着色与其他正式业务可视化颜色口径。
25. 优先覆盖主工作台骨架、顶部栏、左侧工具条、右侧配置区、工具箱、帮助、设置、弹窗/菜单/提示反馈、世界叠加信息等高频界面区域。
26. 允许分阶段落地，但每一阶段都必须避免出现长期不可控的新旧界面混搭。
27. 与界面重构直接相关的底层清理工作，可回链到 `S5-RQ-003`，但界面目标与验收归属本需求。
28. 当前已新增并实现一个文档建档时未单列的新口径：语言设置迁移到左侧底部“设置” dialog 中统一管理。
29. 当前已新增并实现一个文档建档时未单列的新口径：右侧面板在“基地选择”“当前选中”之外，增加独立“电力 / 运行设置”区域，并作为首版工作台的一部分保留。
30. 对“放置”视图内设备分类进一步做二级折叠的建议，当前已被用户明确拒绝；首版继续保持设备分组直接展开展示，不再将“分类可折叠”作为验收要求。
31. 右侧面板当前形态已被用户明确认可；除后续出现新的明确需求外，不再主动追加右侧面板结构性改造目标。
32. 原先“首版主题 UI 落地但不开放真实切换”的限制，已被实际实现提前突破；当前代码已支持 `Ayu Dark` / `Ayu Light` 的真实切换，文档以实际交付为准。
33. 主工作台历史上的零散响应式特例（包括移动页面状态检测及影响主工作台布局的相关特判 CSS / JS）已在 2026-03-11 统一移除；当前工作台一律按“无限宽桌面布局”渲染。帮助页等局部内容区允许保留少量非工作台级的窄宽度排版适配。
34. 左侧主工具条交互口径进一步对齐 VSCode：当左侧面板已展开且再次点击当前激活工具时，应收起左侧面板；当左侧面板已收起时，点击任一主工具应先展开左侧面板并切换到对应 View。
35. 当前已新增一条移动端画布交互需求：支持在主画布上使用双指缩放，并允许在双指操作过程中同时拖动画布；该能力只增强移动端可操作性，不构成响应式布局调整。
36. 左侧工具条图标当前应优先使用 SVG，而不是图标字体，以保证在主题切换、缩放和后续细节调校时具备更稳定的清晰度、一致性与可控性；当前已沿用 SVG 方案并放大主工具图标尺寸。
37. 左侧面板在“已收起后再次点击同一主工具”时，只应展开面板，不应重置该工具的子状态，例如 `place` 视图下的当前操作、当前设备或 `blueprint` 视图下的当前选择。
38. PC 浏览器下页面默认应无全局横向或纵向滚动条；工作台需像此前一样自适应恰好占满视口，滚动只发生在各自面板或画布内部，而不是浏览器页面本身。
39. `Ayu Light` 下允许单独弱化画布网格线，并将管道 / 传送带放置预览从亮白辅助线改为更深的深色系预览，以避免浅色主题下的辅助元素过于刺眼。
40. 所有高频 dialog 在 `Ayu Dark` / `Ayu Light` 下都应保持可读且一致，不允许继续残留固定白字、固定深底等破坏浅色主题的历史样式。
41. 端口优先级组当前改为 `1-9`；点击端口组号时不再使用纵向下拉菜单，而是弹出 `3x3` 的数字 Pad 供直接选择，以避免下拉层被压住或裁切。
42. 删除模式中“单格 / 整条”属于模式二选一，而非独立开关；当前应统一使用单行分段按钮，而不是 `Switch`，以避免占两行且降低可读性。
43. 供电范围当前除描边外，还允许对覆盖到的格子加入一层很浅的蓝色底色，用于提升可读性，但不得压过设备本体与其他核心覆盖层。
44. 删除模式下的“整条”判定当前必须只对真正的 `belt` / `pipe` 轨道生效，不再以 `isBeltLike` / `isPipeLike` 这类 family 级谓词将分流器、汇流器、桥接器等设备一并纳入起点判定。
45. 工具箱中的“设备一览 / 物品一览”当前允许继续按更高信息密度收口；右侧配方卡片不应固定占满整行，而应采用可按内容换行排列的紧凑资料卡布局。物品名称在上一轮基础上再比左侧列表小一档，并同步使用数量 badge、小型机器胶囊和更紧的留白 / 换行规则，避免右侧详情显著大于左侧列表。
46. 自 2026-03-11 起，帮助页的内容重写、用户引导结构、截图 / 动图策略与快捷键页信息架构，独立迁移到 `S5-RQ-010` 维护；本需求仅保留帮助入口位置、dialog 容器形态与工作台级主题一致性职责。
47. 公共蓝图当前允许带一个可选的描述字段；若描述存在，则在左侧蓝图 View 中选中该公共蓝图后，于设备数下方显示该说明；若描述为空，则不占位、不显示额外文案。

## 验收标准

1. 主工作流涉及的核心界面区域已统一到同一套布局与视觉语言。
2. 主要高频操作路径在新界面下可完整完成，且无明显效率退化。
3. 顶栏可通过类似 VSCode 标题栏的 toggle button 显式控制左侧与右侧面板显隐，不再依赖旧抽屉激活逻辑。
4. 左侧工具条稳定提供“放置 / 删除 / 蓝图”3 个入口，图标与文字标签清晰可辨。
5. 点击左侧不同工具按钮时，会像 VSCode 左栏切换 View 一样切换到对应模式的子 Panel，而不是仅切换一个裸模式状态。
6. 当左侧面板处于展开状态时，点击当前已激活的主工具按钮，会像 VSCode 一样直接收起左侧面板。
7. 当左侧面板处于收起状态时，点击任意主工具按钮，会重新展开左侧面板并切换到对应 View。
8. 当左侧面板处于收起状态且点击的是当前已激活的同一个主工具按钮时，只展开面板，不重置该工具的子状态。
9. 左侧 3 个 View 的首版内容分配遵循现有左侧 Panel 结构：
