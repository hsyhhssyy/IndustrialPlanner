# Stage5 README / Project Overview

> 状态：Active（启动于 2026-03-09）

## 文档内容范围、结构与格式约定（必读）

### 1. 本文档负责什么

本文件同时承担 `README` 与 `project-overview` 两个职责，是 Stage5 的唯一总入口。

本文档只负责：

1. 说明 Stage5 的阶段目标、范围、非目标与优先级。
2. 规定 Stage5 文档体系的用途、边界、更新规则与记录格式。
3. 提供所有需求文档、FAQ 文档、索引文档、全局 bug 文档的导航入口。
4. 提供项目级进度视图，帮助 AI 和人类快速判断当前应该做什么。

本文档不负责：

1. 记录某个具体需求的完整拉扯过程。
2. 记录逐轮聊天原文。
3. 记录细粒度测试步骤、 bug 修复细节或验收争议的全部上下文。

### 2. 本文档固定结构

本文档必须长期保持以下结构：

1. 文档内容范围、结构与格式约定
2. Stage5 背景
3. Stage5 总目标
4. 范围与非目标
5. 需求优先级与阶段进度
6. 文档体系说明
7. 更新规则
8. 当前阶段总览

新增内容应追加到既有章节中，不应随意改变大结构。

### 3. 本文档记录格式

1. 项目级结论优先使用短句与编号列表。
2. 需求状态统一使用：`todo`、`in-progress`、`blocked`、`done`。
3. 涉及具体需求时，必须引用对应需求文档，而不是在本文档展开细节。
4. 涉及 AI 协作规则时，先写入 [faq.md](faq.md)，再在本文档中只保留摘要性入口。
5. 涉及每轮对话沉淀时，只记录“形成的决策/变更/问题”，不记录原始聊天流水。

## Stage5 背景

- Stage4 已结束，Stage5 为当前唯一活跃阶段。
- 项目已经进入“继续开发 + 持续清理 + AI 协作沉淀”的阶段。
- Stage5 文档体系已从“按阶段管理模板拆分”改造为“按 AI 工作流组织”：
  - 一个总览文档；
  - 一个需求索引文档；
	- 一个 FAQ 文档；
	- 一个全局 bug / 中断记录文档；
	- 每个需求一个独立文档，采用“当前状态 + 时间线”混合结构，用于沉淀对话记忆、需求变更、验收、测试与 bug 修复记录。

## Stage5 总目标

1. 修复现有高优先级 bug，提升主流程稳定性。
2. 切换到真实游戏 1.1 版本配方与正式图标。
3. 移除“超时空模式”及其过渡逻辑。
4. 在不脱离业务目标的前提下，完成一批高价值代码清理与适度拆分。
5. 交付以下 Stage5 功能：
	- 仿真运行时动态调整能耗；
	- 物品准入口；
	- 起死回生机公共蓝图；
	- 电力 / 电池折线图。
6. 建立由 AI 可持续执行的 Playwright CLI 自动化测试流程。

## 范围与非目标

### 范围

1. 缺陷修复与稳定性收敛。
2. 真实 1.1 数据与正式图标替换。
3. 与 Stage5 主线直接相关的代码清理与模块拆分。
4. Stage5 已确认新功能实现。
5. 自动化测试与交付沉淀。

### 非目标

1. 不进行与主线无关的大型玩法扩展。
2. 不继续扩展“超时空模式”；该模式在 Stage5 中应被移除。
3. 不进行脱离当前需求的全面重写。
4. 不将原始对话逐字存档为主要文档内容。

## 文档体系说明

### 文档清单

1. [README.md](README.md)
	- Stage5 总入口与项目总览。
2. [requirements-index.md](requirements-index.md)
	- 需求索引、状态面板、优先级与链接总表。
3. [faq.md](faq.md)
	- AI 高频澄清问题、默认规则、跨需求共用约定。
4. [bugs.md](bugs.md)
	- 跨需求 bug、中断问题、临时未归属问题的统一入口。
5. [requirements/REQ-001-bug-and-stability.md](requirements/REQ-001-bug-and-stability.md) 至 [requirements/REQ-008-playwright-automation.md](requirements/REQ-008-playwright-automation.md)
	- 每个需求一个独立文档。

### 单需求文档的职责

每个需求文档都用于承载该需求上的完整“结构化记忆”，并采用“状态视图 + 时间线附录”的混合结构，包括：

1. 当前目标、当前口径、当前状态；
2. 当前方案、验收标准、下一步动作；
3. 需求内 bug / 修复记录；
4. 与本需求相关的中断记录；
5. 历次变更与拉扯结论时间线；
6. 测试过程与结果；
7. 待确认问题。

### 全局 bug 文档的职责

[bugs.md](bugs.md) 用于承载不适合直接归入单个需求文档的问题，典型包括：

1. 开发 A 需求过程中暴露出的 B 需求 bug；
2. 暂时无法确认归属的历史遗留问题；
3. 跨需求共用链路上的缺陷；
4. 会打断当前开发节奏的中断事件。

## 更新规则

1. 每次和 AI 讨论后，不要求保留对话原文；只沉淀有效结果。
2. 若结果只影响单个需求，更新对应需求文档。
3. 若结果会影响多个需求或形成默认规则，更新 [faq.md](faq.md)。
4. 若结果是跨需求 bug、临时未归属 bug 或中断事件，先更新 [bugs.md](bugs.md)，再补充相关需求文档。
5. 若结果会改变总体优先级、阶段目标或完成状态，更新 [README.md](README.md) 与 [requirements-index.md](requirements-index.md)。
6. 若某个需求完成验收，必须同步更新：
	- 对应需求文档；
	- 若涉及全局 bug，更新 [bugs.md](bugs.md)；
	- [requirements-index.md](requirements-index.md)；
	- 后续需要时，再补发版摘要。

3. 跨需求 bug 和中断事件优先登记到 [bugs.md](bugs.md)，再回链到相关需求。
4. 单个需求文档既要能快速展示“当前状态”，也要能追溯“时间线过程”。
5. FAQ 只记录跨需求复用的规则，不记录单需求特殊讨论。
