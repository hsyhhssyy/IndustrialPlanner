## 背景与目标

当前 Stage6 已引入操作历史面板与撤销 / 重做能力，但历史仅保存在内存中，刷新页面后会完全丢失。该需求的目标是在不显著抬高复杂度的前提下，为当前布局历史提供“刷新后仍可撤销 / 重做”的能力，同时明确版本升级时不做历史迁移，直接丢弃旧版本历史，避免兼容成本扩散。

在实现过程中，本需求进一步收口为一整套操作历史体验，而不只是“把历史存下来”：历史入口统一放入左侧竖向工具条，时间线支持跳转与粗粒度摘要，键盘快捷键覆盖撤销 / 重做，并且既然删除已经可撤销，则删除类操作不再弹确认框打断流程。

## 当前口径

1. 操作历史按基地分别持久化。
2. 刷新页面后，如果应用版本未变化，应恢复该基地的撤销 / 重做历史。
3. 如果历史存储包裹中的版本号与当前 `APP_VERSION` 不一致，则直接清空历史，不做迁移。
4. 历史入口位于左侧竖向工具条的“操作历史”视图，不再在顶部新增单独按钮。
5. 支持 `Ctrl/Cmd+Z` 撤销、`Ctrl+Shift/Cmd+Shift+Z` 或 `Ctrl+Y` 重做，并全局拦截 `Ctrl/Cmd+S` 使其不触发浏览器保存行为。
6. 历史面板包含“操作”和“历史记录”两个折叠段；历史记录按最新优先显示，可点击跳回任一步。
7. 历史摘要当前基于布局快照差异粗略生成，可区分放置、删除、移动、旋转、配置调整与部分蓝图/物流场景，但不保证达到精细操作日志级别。
8. 既然删除已经纳入撤销 / 重做，删除所有、清空工业区、框选删除等流程默认不再弹确认框。
9. 仿真运行期间，操作历史视图与其他编辑模式一致，统一视为不可用入口。
10. 历史快照仍以布局快照为核心，不在本需求内引入精细操作元数据。

## 当前实现摘要

1. 历史按基地分别维护，每个基地保存 `past`、`current`、`future` 语义上的快照序列。
2. 当前条数上限为单侧 `100` 条，即最多保留 `100` 条撤销历史与 `100` 条重做历史。
3. 历史视图位于左侧活动栏，分为“操作”和“历史记录”两段；时间线按最新优先显示，并标出当前项。
4. 每条历史记录展示粗粒度摘要与设备数，摘要来源于相邻布局快照 diff。
5. 删除所有设备、删除所有物流、清空工业区、框选删除等路径已去除确认弹窗，统一依赖撤销恢复。
6. 仿真开始时若当前停留在“操作历史”视图，会自动切回当前编辑模式；仿真期间按钮禁用。

## 验收标准

1. 页面刷新后，当前基地仍可撤销 / 重做已有历史。
2. 版本号变化后，旧历史不会被恢复。
3. 不影响现有布局持久化与布局读取逻辑。
4. 历史面板、撤销、重做、跳转操作在刷新后仍可正常工作。
5. 删除相关操作不再弹出“不可撤销”类确认提示，且可通过历史恢复。
6. 仿真运行期间，历史入口与操作按钮保持禁用。
