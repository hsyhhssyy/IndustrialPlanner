## 2. 需要补齐的歧义与本文件裁定

### 2.1 输入仲裁粒度

**问题**：`orderedInLinks` 与 `sequential` 的执行先后/粒度不够明确。

**裁定**：

1. 先在组内运行 `orderedInLinks`，得到“本轮端口尝试序列”。
2. 对每个端口候选来料，执行槽位 `sequential` 扫描（按 `slotIndex` 升序）。
3. 命中首个可接收槽位即形成“候选传输计划”，不再继续扫描后续槽位。

### 2.2 RR 游标推进时机

**问题**：未选中/失败时是否推进游标未统一。

**裁定（默认）**：

- 输入游标与输出游标仅在“提交成功”后推进。
- 规划命中但提交失败，不推进。
- 未命中（无候选）不推进。
- 槽位全端口不可达导致跳槽位，本质属于“未成功提交”，不推进。

> 如后续改为“尝试即推进”，必须同步更新本文件的测试矩阵与回归基线。

### 2.3 多缓冲组与端口共享

**问题**：单设备多组场景下是否允许共享物理端口未明确。

**裁定（默认）**：

- 设计上不鼓励端口跨组共享；若出现共享，必须先做“端口所有权仲裁”（每 tick 每端口最多归属一个组）。
- 未完成端口所有权仲裁前，不得直接并行执行组内仲裁。

### 2.4 分流器来源优先的公平性

**问题**：严格优先可能导致传送带来源长期饥饿。

**裁定（默认）**：

- 当前版本保持“硬优先”：分流器来源可用时禁止选择传送带来源。
- 饥饿视为“已知行为”，非 bug；若业务要求避免饥饿，需新增配额/退避策略需求编号。

## 3. 执行模型：Plan / Commit 两阶段

## 3.1 原则

- `Plan` 阶段：只读快照，不修改世界状态。
- `Commit` 阶段：基于计划写入状态；仅提交成功才允许推进上游进度、扣减槽位库存。

## 3.2 计划对象（建议字段）

- `transferId`：全局唯一（单 tick 内唯一即可，建议带 tick 前缀）。
- `tick`：计划产生时刻。
- `fromLinkId` / `toDeviceId` / `toPortId`
- `itemId` / `amount`（当前 amount 固定为 1）
- `bufferGroupId` / `slotIndex`
- `snapshotVersion`：目标槽位与目标端口相关状态版本。

## 3.3 提交校验

提交前必须校验：

1. `snapshotVersion` 未变化；
2. 目标端口当前可接收（含 item-aware 判定）；
3. 槽位仍有可扣减库存（输出）或可写入容量（输入）；
4. 同一 `transferId` 未被提交过（幂等保障）。

任一失败即提交失败，不做部分写入。

## 4. 关键不变量（Invariants）

每个 tick 结束后必须成立：

1. 槽位容量约束：`0 <= amount <= capacity`。
2. 物品一致性：非空槽位必须有 `currentItemId`；空槽位可为 `null`。
3. 推进守恒：未提交成功的上游链路不得推进。
4. 扣减守恒：成功提交次数 = 实际库存扣减次数（逐槽位可对账）。
5. 单传输唯一性：同一 `transferId` 最多成功一次。

## 5. RR 游标推进矩阵（执行契约）

| 场景 | 计划命中 | 提交成功 | 输入游标推进 | 输出游标推进 | 上游进度推进 |
|---|---:|---:|---:|---:|---:|
| A. 命中且成功 | 是 | 是 | 是 | 是 | 是 |
| B. 命中但提交失败 | 是 | 否 | 否 | 否 | 否 |
| C. 未命中（无可行候选） | 否 | 否 | 否 | 否 | 否 |
| D. 槽位可选但全端口不可达 | 否（可执行意义） | 否 | 否 | 否 | 否 |

> 若后续产品决策调整“失败也推进游标”，需改表并补充 anti-starvation 证明与回归用例。

## 6. 槽位扫描与端口扫描规范

### 6.1 输入

- 槽位扫描顺序固定：`slotIndex` 从小到大。
- 命中首个可接收槽位即停止。
- `free`：空槽可收，非空仅同类可叠加。
- `pinned`：仅 `itemId == pinnedItemId` 可收（即使空槽也不接收其他物品）。

### 6.2 输出

1. 槽位优先：先前槽位，后后槽位。
2. 对候选槽位，按 `orderedOutLinks` 顺序尝试端口。
3. 若该槽位全端口不可达，跳到下一槽位。
4. 命中首个“槽位可出 + 端口可达”的组合后输出并结束。

## 7. Bug 防线设计（对应高风险）

### 7.1 幽灵推进

- 规则：上游推进动作只能在 `CommitSuccess(transferId)` 回调中触发。
- 禁止在 `Plan` 或“尝试发送”阶段推进。

### 7.2 RR 漂移

- 游标只允许在单一函数内修改（推荐 `advanceCursorOnCommitSuccess`）。
- 任何其他路径修改游标视为实现缺陷。

### 7.3 重复扣减 / 丢物

- `transferId` 幂等提交：重复提交直接拒绝或 no-op。
- 提交写入采用原子序列：校验 -> 扣减/写入 -> 标记成功。

### 7.4 顺序偏差与边界错误

- 协议储存箱固定 6 槽：扫描索引必须覆盖 `[0..5]`。
- 单槽容量固定 50：不得出现 `amount=51`。
- 输入不得将“空位与同类同权”实现成“优先同类堆叠”。

## 8. 最小验收用例（G/W/T）

## 8.1 推进守恒

- Given：两个上游竞争同一下游，只有一路本 tick 被选中。
- When：运行 1 tick。
- Then：未选中链路输送进度不前移。

- Given：链路被选中，但下游 item-aware 拒收导致提交失败。
- When：运行 1 tick。
- Then：该链路进度不前移，库存不扣减。

## 8.2 槽位顺序

- Given：第 1 槽有货但全端口不可达，第 2 槽有货且端口可达。
- When：执行输出决策。
- Then：跳过第 1 槽，从第 2 槽输出 1 个。

- Given：第 1 槽空，第 2 槽为同类未满。
- When：有来料输入。
- Then：应命中第 1 槽（首个可接收槽位），不跳到第 2 槽。

## 8.3 RR 一致性

- Given：固定拓扑与固定随机种子。
- When：连续运行 N tick（建议 N>=200）。
- Then：每次重放结果一致，游标轨迹一致。
