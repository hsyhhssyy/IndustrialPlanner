# Stage4 专项重构规格：下游拉取（Pull-driven）仿真模型

> 状态：Draft（待执行）
> 日期：2026-03-07
> 负责人：GitHub Copilot（方案草拟）
> 触发背景：S4-RQ-004 现行实现在多设备竞争场景下仍存在“发送端先选 + 接收端被动门禁”的语义偏差，无法在同 tick 内稳定表达“接收端重选候选”的期望行为。

---

## 1. 背景与问题定义

当前实现虽然采用 `Plan -> Commit` 两段结构，但 `Plan` 阶段仍以发送端（source）遍历出边为主，接收端仅返回可接收状态。这会导致以下问题：

1. **仲裁起点错误**：输入优先级（如 A/B/C）是接收端规则，但决策由发送端先发起，导致规则表达不自然。
2. **同 tick 回退能力弱**：当接收端首选上游在后续仲裁中被其他下游占用时，无法在同 tick 自动回退到次优候选。
3. **遍历顺序敏感**：先遍历到谁会影响计划形成，影响确定性和可解释性。
4. **历史语义残留**：`canTry/canAccept` 混合了旧时序门禁与新 item-aware 判定，职责边界不清。

---

## 2. 本次重构目标（一次全量）

本次采用一次性全量改造，不做 Phase 灰度，目标如下：

1. 将仿真物流规划模型切换为 **Pull-driven（下游拉取）**。
2. 接收端负责“我要从谁拉”的优先级决策；发送端负责“我实际给谁”的冲突裁决。
3. 在单 tick 内通过固定点迭代（fixed-point）完成冲突消解与候选回退，不依赖设备遍历顺序。
4. 删除 `canTry/canAccept` 语义，替换为更直接的“请求可行/提交可行”判定。
5. 保持推进守恒：仅提交成功才推进游标、推进输送进度、扣减库存。

---

## 3. 非目标（本次不做）

1. 不改 UI 展示层与交互。
2. 不改设备配方系统、电力系统、生产周期系统。
3. 不引入额外调度策略（如随机、权重学习、配额平衡）。

---

## 4. 新执行模型（每 tick）

### 4.1 总流程

每 tick 执行以下 4 步：

1. **Build Receiver Intents**：所有可接收设备基于输入规则生成拉取意图列表（按优先级排序）。
2. **Resolve by Sender Arbitration**：按发送端输出能力、输出规则，对收到的拉取意图做裁决并形成候选匹配。
3. **Fixed-point Re-resolve**：对未满足意图自动回退到下一候选并重试，直到稳定（无新增匹配/无状态变化）或达上限。
4. **Atomic Commit**：统一提交匹配结果；成功后推进游标与状态，失败不产生副作用。

### 4.2 关键语义

- **输入规则属于接收端**：端口优先级、分层优先（分流器 > 传送带）、输入 RR、item 过滤由接收端决定。
- **输出规则属于发送端**：输出 RR、槽位顺序、端口可达性、单 tick 可发额度由发送端决定。
- **竞争裁决由发送端完成**：同一发送端若收到多个下游请求，由发送端按自身规则选中可服务请求。

## 6. 冲突与回退规则（必须满足）

### 6.1 接收端回退

若接收端首选意图未被发送端选中，则该接收端在同 tick 下一轮 fixed-point 中尝试下一个候选。

### 6.2 发送端唯一供给

同一发送端同一可发送单元（槽位或端口可发单元）在同 tick 最多服务一个匹配。

### 6.3 去遍历顺序依赖

结果不得依赖设备遍历先后。相同输入状态应得到相同输出匹配集合。

### 6.4 提交失败处理

提交前再次校验接收可行性与源可扣减性。失败则该匹配作废，不推进游标、不推进进度、不扣库存。

---

## 7. 游标推进契约（替代现有口径）

### 7.1 输入游标推进

仅当接收端匹配提交成功后，推进对应缓冲组 `inCursor`。

### 7.2 输出游标推进

仅当发送端匹配提交成功后，推进对应缓冲组 `outCursor`。

### 7.3 失败与未命中

- 未命中：不推进。
- 命中但提交失败：不推进。

## 8. API/接口变更要求

### 8.1 删除（或废弃）

- 旧 `ReceiveState` 三态语义：`canTry/canAccept`。
- 旧“发送端先选链路”的主计划入口语义。

### 8.2 新接口方向

- `buildReceiverPullIntents(...)`
- `resolvePullIntentsBySenders(...)`
- `solvePullMatchesFixedPoint(...)`
- `commitPullMatches(...)`

### 8.3 保留复用

- 缓冲组与槽位运行时结构（`bufferGroups`, `slots`, `inCursor`, `outCursor`）保留。
- 提交阶段“成功才推进”的核心约束保留。

## 10. 不变量（Invariants）

每 tick 结束必须满足：

1. 守恒：成功提交数 = 源扣减数 = 目标接收数。
2. 容量：所有槽位 `0 <= amount <= capacity`。
3. 唯一：同一 `transferId` 最多成功一次。
4. 推进守恒：未提交成功链路不推进。
5. 确定性：相同输入快照下，匹配与提交结果一致。

## 11. 验收用例（最小集）

### 11.1 用户提出冲突场景（必须通过）

- 设备1输入端口优先级 A > B > C，本轮首选 B。
- 设备2输出端口优先级 D > E > F，但最终将可发单位分配给 F（与设备1的 B 不匹配）。
- 要求：设备1在同 tick 的 fixed-point 后续轮次自动回退到 C 并完成拉取（若 C 可行）。

### 11.2 多下游竞争同上游

- 两个下游同时请求同一上游同一可发单元。
- 要求：仅一个请求成功，另一个请求在同 tick 回退候选或失败结束；无重复扣减。

### 11.3 分层优先

- 汇流器存在可用分流器来源和可用传送带来源。
- 要求：只允许选择分流器来源。

### 11.4 提交失败无副作用

- 计划命中后提交前状态变化导致失败。
- 要求：游标不推进、库存不变化、进度不前移。

---

## 12. 性能与稳定性约束

1. fixed-point 迭代上限：`maxRounds = max(8, totalLinks)`。
2. 超上限处理：记录告警并停止新增匹配（保持正确性优先）。
3. 复杂度控制：候选列表与匹配表采用 map/set 去重，避免 N² 无界增长。
