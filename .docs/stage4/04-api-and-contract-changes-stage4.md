# Stage4 API 与契约变更

## 1. 契约范围

- Stage4 当前涉及展示层契约补齐（S4-RQ-001）、协议储存箱行为契约补齐（S4-RQ-002）、下游接收前置判定 item-aware 契约补齐（S4-RQ-003）与统一缓冲组契约重构（S4-RQ-004）。
- 不新增仿真状态码。

## 2. 展示契约

- 设备详情需读取并展示 `DeviceTypeDef.powerDemand`。
- 展示单位统一为 `kW`。
- 发电设备显示 `powerDemand=0`。

## 3. 协议储存箱行为契约

- 协议储存箱支持仿真开始时置入初始物品。
- 协议储存箱固定为 6 格，每格容量 50。
- 输出契约：按第 1 格到第 6 格顺序检索，单次输出 1 个。
- 输入契约：按第 1 格到第 6 格顺序检索，空位与同类同权，命中首个可放置格即放入。
- 实现机制：由 S4-RQ-004 的统一缓冲组与槽位规则承载。

## 4. 下游接收前置判定契约（S4-RQ-003）

- `canReceiveOnPortWithPlan` 需支持透传待传输物品ID（item-aware）。
- 前置判定返回的 `canTry/canAccept` 需可反映目标设备对该物品的接收能力。
- 本契约独立于物品准入口/管道准入口功能需求，单独验收。

## 5. 统一缓冲组与槽位规则契约（S4-RQ-004）

- 单设备可配置多组端口-缓冲区映射；每组独立维护输入仲裁状态。
- 每组输入使用统一 `orderedInLinks`（RR）生成端口尝试顺序。
- 每组输出使用统一 `orderedOutLinks`（RR）生成端口尝试顺序。
- 槽位模式支持 `free/pinned`：
	- `free`：空位可接收；非空仅同类堆叠。
	- `pinned`：仅接受绑定物品，即使空位也不接受非绑定物品。
- 槽位接收按 `sequential` 检索，命中首个可接收槽位即写入。
- 槽位输出按顺序检索：前槽位先尝试；对候选槽位按 `orderedOutLinks` 逐端口判定；若该槽位全端口不可接收，则跳到下一槽位。
- 汇流器输入契约：`orderedInLinks` 需支持“分流器来源优先于传送带来源”；存在可用分流器来源时禁止选择传送带来源，仅在分流器来源不可用时回退。
- 推进守恒契约：上游输送进度推进必须以后续传输提交成功为前提；未被仲裁选中或提交失败时不得推进。

## 6. 破坏性变更

- 当前无预期 Breaking 变更。
