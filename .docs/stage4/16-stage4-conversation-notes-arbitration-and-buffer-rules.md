# Stage4 会话纪要：仲裁与缓冲组规则（2026-03-05）

## 1. 纪要目的

- 记录本轮需求澄清与设计决策。
- 将“输入/输出仲裁、缓冲槽位规则、推进守恒”形成统一实现口径。
- 明确用户当前核心担忧：实现阶段可能出现多轮仲裁下的错误推进问题。

## 2. 已确认需求与关键结论

### 2.1 S4-RQ-002 协议储存箱六格库存化

- 协议储存箱支持仿真开局置入初始物品。
- 固定 6 槽位，每槽容量 50。
- 输入按槽位顺序检索，空位与同类同权（不做额外优先堆叠）。
- 输出按槽位顺序进行。
- 本需求实现机制挂靠 S4-RQ-004。

### 2.2 S4-RQ-003 前置判定 item-aware 化

- `canReceiveOnPortWithPlan` 需要透传物品ID。
- 前置判定需反映“对该物品”的可接收能力。
- 该需求为独立需求，不作为准入口需求附属。

### 2.3 S4-RQ-004 统一缓冲组与槽位顺序规则重构

- 单设备可配置多组端口-缓冲区映射。
- 每组独立维护仲裁状态（输入/输出分别有 RR 游标）。
- 槽位模式：`free` / `pinned`。
- 输入统一：`orderedInLinks` + `sequential`（首个可接收位）。
- 输出统一：槽位顺序优先 + `orderedOutLinks` 端口顺序。
- 当候选槽位在全部输出端口不可达时，允许跳到下一槽位继续判定。

## 3. 汇流器优先级策略（配合 S4-RQ-004）

- 汇流器输入侧：分流器来源端口优先于传送带来源端口。
- 当存在至少一个“可输入的分流器来源端口”时，不得选择传送带来源端口。
- 仅当分流器来源全部不可输入时，才回退选择传送带来源端口。
- 在实现层对应为 `orderedInLinks` 的分层优先策略（来源类型分层优先 > 层内 RR）。

## 4. 关键风险与硬约束

### 4.1 用户提出的关键风险

- 在多上游竞争下，若未选中上游仍持续推进进度，可能反复“白推进”并最终错误推进到终点。

### 4.2 已确认的硬约束（推进守恒）

- 上游仅在“本轮被仲裁选中且传输提交成功”时允许推进输送进度。
- 未被选中或提交失败时，必须保持原进度，不得前移。
- 该约束属于功能正确性要求，不是性能优化项。

## 5. 对当前仿真执行模型的共识

- 当前 tick 内存在多轮规划/仲裁（循环收敛），再统一提交传输。
- 因此实现 S4-RQ-004 时，必须确保“仲裁阶段”与“进度推进阶段”的因果关系受推进守恒约束保护。

## 6. 未决实现疑问（待后续设计定稿）

- 采用“先仲裁后推进”还是“先推进后回滚”的实现策略。
- 纪要建议：优先采用“先仲裁后推进”，以降低多轮仲裁下的状态回滚复杂度与错位风险。

## 7. 落地建议（开发前检查）

1. 在实现文档中单列“推进守恒”校验点与失败示例。
2. 为多轮仲裁构造压力用例：
   - 高优先级上游间歇可用；低优先级上游持续有货。
   - 下游准入口过滤导致候选端口间歇拒收。
3. 验证未被选中链路在连续多 tick 下进度保持不变。
4. 验证提交失败链路在连续多 tick 下不会推进到终点。
