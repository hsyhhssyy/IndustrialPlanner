# Stage 文档模板 / README / Project Overview 模板

> 用途：所有新 Stage 必须从本模板复制开始，不允许跳过模板直接手写零散文档。

## 文档内容范围、结构与格式约定（必读）

### 1. 本文档负责什么

本文件同时承担 `README` 与 `project-overview` 两个职责，是一个 Stage 的唯一总入口。

本文档只负责：

1. 说明当前 Stage 的阶段目标、范围、非目标与优先级。
2. 规定当前 Stage 文档体系的用途、边界、更新规则与记录格式。
3. 提供所有需求文档、FAQ 文档、索引文档、全局 bug 文档的导航入口。
4. 提供项目级进度视图，帮助 AI 和人类快速判断当前应该做什么。

本文档不负责：

1. 记录某个具体需求的完整拉扯过程。
2. 记录逐轮聊天原文。
3. 记录细粒度测试步骤、 bug 修复细节或验收争议的全部上下文。

### 2. 本文档固定结构

本文档必须长期保持以下结构：

1. 文档内容范围、结构与格式约定
2. Stage 背景
3. Stage 总目标
4. 范围与非目标
5. 需求优先级与阶段进度
6. 文档体系说明
7. 更新规则
8. 当前阶段总览

### 3. 本文档记录格式

1. 项目级结论优先使用短句与编号列表。
2. 需求状态统一使用：`todo`、`in-progress`、`blocked`、`done`。
3. 涉及具体需求时，必须引用对应需求文档，而不是在本文档展开细节。
4. 涉及 AI 协作规则时，先写入 [faq.md](faq.md)，再在本文档中只保留摘要性入口。
5. 涉及每轮对话沉淀时，只记录“形成的决策/变更/问题”，不记录原始聊天流水。

## 文档体系说明

### 文档清单

1. [README.md](README.md)
   - Stage 总入口与项目总览。
2. [requirements-index.md](requirements-index.md)
   - 需求索引、状态面板、优先级与链接总表。
3. [faq.md](faq.md)
   - AI 高频澄清问题、默认规则、跨需求共用约定。
4. [bugs.md](bugs.md)
   - 跨需求 bug、中断问题、临时未归属问题的统一入口。
5. [requirements/REQ-template.md](requirements/REQ-template.md)
   - 单需求文档模板；实际使用时复制并改名。

### 单需求文档的职责

每个需求文档都用于承载该需求上的完整“结构化记忆”，并采用“状态视图 + 时间线附录”的混合结构，包括：

1. 当前目标、当前口径、当前状态；
2. 当前方案、验收标准、下一步动作；
3. 需求内 bug / 修复记录；
4. 与本需求相关的中断记录；
5. 历次变更与拉扯结论时间线；
6. 测试过程与结果；
7. 待确认问题。

### 全局 bug 文档的职责

[bugs.md](bugs.md) 用于承载不适合直接归入单个需求文档的问题，典型包括：

1. 开发 A 需求过程中暴露出的 B 需求 bug；
2. 暂时无法确认归属的历史遗留问题；
3. 跨需求共用链路上的缺陷；
4. 会打断当前开发节奏的中断事件。

## 更新规则

1. 每次和 AI 讨论后，不要求保留对话原文；只沉淀有效结果。
2. 若结果只影响单个需求，更新对应需求文档。
3. 若结果会影响多个需求或形成默认规则，更新 [faq.md](faq.md)。
4. 若结果是跨需求 bug、临时未归属 bug 或中断事件，先更新 [bugs.md](bugs.md)，再补充相关需求文档。
5. 若结果会改变总体优先级、阶段目标或完成状态，更新 [README.md](README.md) 与 [requirements-index.md](requirements-index.md)。
6. 若某个需求完成验收，必须同步更新：
   - 对应需求文档；
   - 若涉及全局 bug，更新 [bugs.md](bugs.md)；
   - [requirements-index.md](requirements-index.md)；
   - 后续需要时，再补发版摘要。

## 当前阶段总览

### 当前文档策略

- 当前 Stage 使用“AI 外部记忆优先”的文档模式。
- 文档核心目的不是形式化归档，而是减少重复解释、降低上下文漂移、提升 AI 连续执行质量。

### 当前需求入口

- 需求总表： [requirements-index.md](requirements-index.md)
- 高频规则： [faq.md](faq.md)
- 全局 bug： [bugs.md](bugs.md)
- 需求详情： [requirements](requirements)

### 当前执行原则

1. 先稳定主流程，再做新增功能。
2. 单个需求文档既要能快速展示“当前状态”，也要能追溯“时间线过程”。
3. 跨需求 bug 和中断事件优先登记到 [bugs.md](bugs.md)，再回链到相关需求。
4. FAQ 只记录跨需求复用的规则，不记录单需求特殊讨论。
