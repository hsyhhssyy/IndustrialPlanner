# Stage FAQ / AI 协作规则 模板

> 用途：复制到具体 Stage 后，沉淀 AI 高频反复追问的问题，以及跨需求可复用的默认规则。

## 文档内容范围、结构与格式约定（必读）

### 1. 本文档负责什么

本文档用于沉淀 AI 在当前 Stage 中高频反复追问的问题，以及跨需求可复用的默认规则。

本文档只负责：

1. 记录跨需求共用的默认约定。
2. 记录 AI 容易重复确认的口径、优先级、判断规则与记录方式。
3. 降低后续对话中的重复澄清成本。

本文档不负责：

1. 记录某个单独需求的特殊讨论。
2. 记录完整聊天过程。
3. 替代需求文档中的验收、测试、 bug 修复记录。
4. 替代 [bugs.md](bugs.md) 记录跨需求 bug / 中断条目。

### 2. 本文档固定结构

本文档必须长期保持以下结构：

1. 文档内容范围、结构与格式约定
2. 使用规则
3. FAQ 条目
4. 新增条目模板
5. 更新日志

### 3. 本文档记录格式

每个 FAQ 条目使用以下固定格式：

- `FAQ-ID`：唯一编号，例如 `<StagePrefix>-FAQ-001`
- `问题`：AI 反复会问的真实问题
- `默认答案`：当前默认执行口径
- `适用范围`：影响哪些需求或整个 Stage
- `何时改写`：出现什么条件时需要更新本条

## 使用规则

1. 只有“会跨需求复用”或“AI 已多次重复追问”的问题，才写入 FAQ。
2. 单需求特殊规则写回对应需求文档，不写 FAQ。
3. bug 条目、打断开发的临时问题、未归属缺陷优先写入 [bugs.md](bugs.md)，而不是 FAQ。
4. FAQ 一旦形成默认口径，后续执行优先按 FAQ 处理，除非需求文档明确覆盖。
5. 若 FAQ 与单需求文档冲突：
   - 单需求内的局部特殊口径，以单需求文档为准；
   - 跨需求通用规则，以 FAQ 为准。

## FAQ 条目

### <StagePrefix>-FAQ-001：当前 Stage 文档体系应该如何使用？

- `问题`：当前 Stage 文档到底看哪些？更新时先改哪里？
- `默认答案`：
  1. 先看 [README.md](README.md) 了解总体目标与优先级。
  2. 再看 [requirements-index.md](requirements-index.md) 找到目标需求。
  3. 进入对应需求文档处理该需求的全部细节。
  4. 若形成跨需求规则，再更新本文档。
- `适用范围`：整个 Stage
- `何时改写`：文档结构再次调整时。

### <StagePrefix>-FAQ-002：是否需要记录每轮对话原文？

- `问题`：和 AI 的每轮交流要不要逐字记录？
- `默认答案`：不记录原文，只记录有效结果，包括：新需求、需求变更、已确认决策、被否决方案、验收变化、测试结果、 bug 发现与修复结论。
- `适用范围`：整个 Stage
- `何时改写`：若后续引入专门的对话归档系统时。

### <StagePrefix>-FAQ-003：开发过程中突然冒出的 bug 记在哪里？

- `问题`：开发 A 需求时突然发现 B 需求 bug，或者发现一个暂时说不清归属的历史 bug，应该记在哪？
- `默认答案`：先记入 [bugs.md](bugs.md)。如果后续确认它明显属于某个需求，再在对应需求文档补充影响与处理结果，并在 bug 条目里回链。
- `适用范围`：整个 Stage
- `何时改写`：bug 管理策略改变时。

### <StagePrefix>-FAQ-004：需求文档应该按时间线写，还是按状态写？

- `问题`：单需求文档到底应该以时间线为主，还是以当前状态为主？
- `默认答案`：采用“状态视图 + 时间线附录”的混合结构。文档前半部分让 AI 快速接手当前状态，后半部分保留工作时间线，记录中断、恢复、测试往返与需求拉扯。
- `适用范围`：整个 Stage
- `何时改写`：需求文档模板变化时。
