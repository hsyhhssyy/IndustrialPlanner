### 3.1 S3-RQ-008 产线规划器流程图视图与视角切换

- 状态：`todo`
- 业务价值：
  - 当前树状列表在跨链路理解上可读性不足，流程图可提升依赖关系与瓶颈定位效率。
  - 提供“按产物 / 按设备”两种观察视角，支持不同使用场景下的快速分析。
- 详细需求：
  - 在产线规划器中新增流程图展示，不替代现有树状列表。
  - 流程图支持两种可切换视角：
    - `byProduct`：以产物流转为主线组织节点与连线。
    - `byBuilding`：以设备/建筑为主线组织节点与连线。
  - 用户可在同一结果上下文中切换视角，切换后保持同一组规划输入与计算结果，不触发重算。
- 非目标（可选）：
  - 本需求不包含自动布局算法优化与美化主题扩展。
  - 本需求不改变现有配方求解逻辑与结果数值。
- 影响面（domain/sim/ui/data/test）：
  - `ui`：新增流程图容器、节点渲染、视角切换控件。
  - `domain`：补充流程图投影视图模型（按产物/按设备映射）。
  - `data`：新增视图偏好持久化键（如当前视角）。
  - `test`：新增两种视角渲染一致性与切换稳定性回归。
  - `sim`：无直接变更（只读消费既有结果）。
- 依赖项：
  - 规划器结果结构需可稳定映射为节点与边。
  - 既有树状列表数据源需复用，避免双口径。
- 风险与回滚点：
  - 风险：双视角映射规则不一致导致信息不对齐。
  - 风险：大规模结果渲染性能抖动影响交互。
  - 回滚点：保留树状列表为默认展示，流程图能力可按开关整体下线。
- 验收标准（DoD）：
  1. 规划器结果页可展示流程图，且与树状列表可并存。
  2. 支持 `byProduct` 与 `byBuilding` 双视角切换，切换无报错且不触发规划重算。
  3. 同一输入下，两种视角与树状列表在关键产物与关键设备集合上信息一致。
  4. 视角选择可持久化，刷新后保持用户上次选择。
  5. Stage2 冻结主流程与 Stage3 已有验收项无 P0/P1 回归。

### 3.2 S3-RQ-009 全设备功耗字段补齐（领域模型）

- 状态：`todo`
- 业务价值：
  - 当前设备功耗数据不完整会导致电力统计与设备级观测失真。
  - 补齐全设备功耗字段可为后续电力计算、可视化与平衡调优提供统一数据基线。
- 详细需求：
  - 在领域模型中为所有设备补齐 `powerDemand`（消耗功率）数据。
  - 要求设备配置中显式可读，不依赖运行时隐式推导。
  - 对不消耗电力设备，`powerDemand` 显式为 `0`。
  - 发电设备在设备属性页展示 `powerDemand=0`。
- 非目标（可选）：
  - 本需求不包含发电设备供电曲线建模细化。
  - 本需求不包含 UI 样式改造。
- 影响面（domain/sim/ui/data/test）：
  - `domain`：设备配置结构与设备注册表数据补齐。
  - `sim`：读取设备功耗口径时改为统一消费 `powerDemand`。
  - `ui`：设备详情与电力看板可消费完整字段。
  - `test`：补充配置完整性校验与关键设备功耗回归。
  - `data`：无新增持久化结构（仅配置数据补齐）。
- 依赖项：
  - 设备清单口径稳定（含武陵与基础区域设备）。
  - 电力字段契约与默认值策略在 Stage3 内达成一致。
- 风险与回滚点：
  - 风险：批量录入功耗值时单位或量纲不一致。
  - 风险：旧设备缺失字段导致运行时 fallback 偏差。
  - 回滚点：保留旧口径兼容读取，按特性开关切换到补齐字段。
- 验收标准（DoD）：
  1. 领域模型设备配置中“所有可放置设备”均具备可解析的 `powerDemand`。
  2. 缺失字段率为 0（按设备注册表全量扫描）。
  3. 不耗电设备口径一致且在文档中明确（`powerDemand=0`）。
  4. 发电设备在属性页显示 `powerDemand=0`。
  5. 电力聚合结果在引入全量字段后与预期对账通过。
  6. Stage2 冻结主流程与 Stage3 已有验收项无 P0/P1 回归。

### 3.3 S3-RQ-010 自动布线（Auto Routing）独立模块

- 状态：`todo`
- 业务价值：
  - 降低产线规划后手动连接成本，缩短从“规划结果”到“可执行布局蓝图”的路径。
  - 将耗时生成过程异步化，避免阻塞前台交互。
  - 通过独立模块 + 任务队列形态，为后续批量生成与调度扩展提供基础。
- 详细需求（已补充）：
  - 新增全新功能模块“自动布线（Auto Routing）”，作为后台任务系统运行。
  - UI 增加“任务队列”入口按钮，位置在“规划器按钮旁边”。
  - 在规划器内部提供“发送到队列”操作：当用户配置好一份规划后，可将该规划提交到后台队列。
  - 后台队列按任务消费规划输入，自动生成一份“摆放蓝图”。
  - 生成蓝图必须可放置在“当前基地尺寸”内；当前阶段仅考虑基地尺寸边界约束。
  - 任务参数必须包含基地尺寸（宽/高或等效边界定义），并作为生成约束输入。
  - 自动布线模块在架构上与其他功能模块解耦，保持独立边界。
  - 自动布线模块只读取领域模型，不依赖其他模块运行态。
  - 任务队列需具备基础状态流转：`queued` → `running` → `done | failed | canceled`。
  - 蓝图生成过程允许长时运行，任务队列需提供可观测进度信息（至少在 `running` 阶段可见进度）。
- 输出定义：
  - 每个成功任务（`done`）必须满足“所有连接均正确路由完成”，并产出一份可被系统识别的“摆放蓝图结果对象”（具体字段后续在 API 文档细化）。
- 非目标（可选）：
  - 不与现有模块形成双向依赖。
  - 不改写既有仿真核心规则。
  - 本需求不包含自动布线算法“最优解保证”。
- 影响面（domain/sim/ui/data/test）：
  - `ui`：
    - 顶部新增任务队列入口（位于规划器按钮旁）。
    - 规划器内新增“发送到队列”触发按钮。
    - 新增任务队列面板（展示任务状态与蓝图产出）。
  - `domain`：读取领域模型实体、配方与连接约束（只读）。
  - `sim`：默认无直接耦合，按独立模块落地。
  - `data`：新增任务队列任务实体与状态持久化（仅会话级；关闭浏览器后全部丢失）。
  - `test`：补充入队、状态流转、失败恢复、蓝图产出一致性、尺寸约束与进度上报测试。
- 依赖项：
  - 领域模型数据口径稳定且可被只读消费。
  - 规划器触发协议、规划快照结构与队列任务载荷格式明确。
  - 基地尺寸参数定义与任务载荷字段命名达成一致。
- 风险与回滚点：
  - 风险：入口在规划器但模块独立，若边界定义不清易形成隐式耦合。
  - 风险：后台任务积压导致长时间等待或状态不一致。
  - 风险：尺寸约束处理不完整导致蓝图越界。
  - 风险：蓝图生成失败时缺乏可诊断信息影响可用性。
  - 回滚点：保持入口与队列面板可开关下线，异常时退化为仅保留规划器原有流程。
- 验收标准（DoD）：
  1. 顶栏存在任务队列入口，且位置在规划器按钮旁边。
  2. 规划器内可将“当前规划”发送到自动布线后台队列。
  3. 任务状态按 `queued` → `running` → `done | failed | canceled` 可观测。
  4. 任务参数包含基地尺寸，且生成结果满足“可放置在当前基地尺寸内”。
  5. `running` 任务可查看进度信息（如百分比或阶段进度）。
  6. `done` 任务必须满足“所有连接均正确路由完成”；`failed` 任务可观测失败状态与失败原因。
  7. 自动布线模块代码结构独立，且计算仅依赖领域模型只读数据。
  8. Stage2 冻结主流程与 Stage3 已有验收项无 P0/P1 回归。
