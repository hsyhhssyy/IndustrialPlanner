# Stage3 自动布线算法设计与解读（定稿口径）

> 说明（2026-03-05）：自动布线需求已在 Stage3 结项时废弃；本文件仅保留为历史设计记录。

> 目标：给出“自动布线（Auto Routing）”的可落地算法方案，作为 Stage3 必做项完成交付，支撑后台队列异步生成“可放置在当前基地尺寸内”的摆放蓝图。

## 1. 问题定义

输入（任务参数）：
- 规划快照（目标产线、设备需求、配方与流向）
- 基地尺寸约束（宽/高或等效边界）
- 领域模型只读数据（设备尺寸、端口、旋转规则、连接规则）

输出：
- 一份可落地摆放蓝图：
  - 所有设备块位置与朝向
  - 连接关系与走线路径
  - 质量指标（线长、拥塞、占地率、未满足连接）

关键特性：
- 设备块允许多放置模板（多锚点/多端口取向）
- 设备块允许旋转（0/90/180/270）
- 布局不强制严格按生产链路拓扑顺序
- 仅约束“可放置 + 可连接 + 代价尽量低”

## 2. 建模思路

### 2.1 设备块状态

对每个设备块 i，定义状态：
- 位置：$(x_i, y_i)$
- 朝向：$r_i \in \{0,90,180,270\}$
- 模板：$m_i \in M_i$（同设备可选模板集合）

模板用于表达：
- 端口布局差异（输入/输出端口位置）
- 占地包围盒差异
- 偏好的连接侧（如更适合主干接入）

### 2.2 连接需求图

从规划结果构造加权有向图 $G=(V,E)$：
- 节点 $V$：设备块或设备块组
- 边 $E$：物流连接需求，边权代表流量/重要性

注意：
- 图用于“优化偏好”，不是严格层级摆放命令。
- 允许打破拓扑层次，只要最终连接可行且代价更优。

## 3. 约束与目标

### 3.1 硬约束（必须满足）

1. 边界约束：设备占地完全位于基地尺寸内。
2. 非重叠约束：设备占地包围盒不可重叠。
3. 连接可行性：每条关键连接至少有可行路由（或被明确标记为失败原因）。
4. 端口合法性：连接必须满足端口类型与方向规则。

### 3.2 软目标（可权衡优化）

最小化目标函数：

$$
J = \alpha L + \beta C + \gamma O + \delta U + \epsilon B
$$

- $L$：总线长（越短越好）
- $C$：拥塞/交叉惩罚（越少越好）
- $O$：端口朝向不友好惩罚
- $U$：未满足流量或绕行惩罚
- $B$：布局碎片化惩罚（过度分散/无效留白）

推荐默认权重（第一版）：
- $\alpha=1.0,\beta=2.0,\gamma=0.8,\delta=5.0,\epsilon=0.6$

解释：先保证可连接与可用，再优化线长和美观。

## 4. 求解框架（先可行，后优化）

### 阶段 A：快速可行解（Feasible First）

目的：尽快产出可放置初解，作为后台任务保底结果。

流程：
1. 对高权重节点优先放置（可按连接度/流量排序）。
2. 对每个节点枚举候选 $(x,y,r,m)$。
3. 过滤硬约束非法候选。
4. 用增量代价函数打分，选择当前最优候选。
5. 每放置一批节点，执行一次轻量路由可达性检查。

输出：
- 一份可行蓝图（质量中等但可用）

### 阶段 B：局部优化（LNS / 模拟退火）

目的：在可行解基础上降低总代价 $J$。

动作集合：
- `move(block, dx, dy)`
- `rotate(block, r')`
- `swap(block_i, block_j)`
- `changeTemplate(block, m')`
- `clusterMove(group)`（强耦合块整体移动）

接受准则：
- 改善解直接接受
- 轻微变差解按温度概率接受（避免局部最优）

终止条件：
- 达到时间预算
- 连续 N 轮无改进
- 用户取消任务

### 阶段 C：重布线与修复

在固定布局上运行网格路由：
- 主路由：A*（含拥塞代价）
- 冲突处理：rip-up & reroute（拆线重路由）

若拥塞持续过高：
- 回到阶段 B 做布局微调，再重路由。

## 5. 后台队列与进度模型

任务状态：
- `queued` -> `running` -> `done | failed | canceled`

`done` 判定口径：
- 必须满足“所有连接均已正确路由完成”。
- 任一连接未满足时不得标记 `done`，应进入 `failed` 并输出诊断信息。

建议进度结构：
- `phase`：`prepare | feasible | optimize | route | finalize`
- `phaseProgress`：当前阶段 0~100
- `overallProgress`：全局 0~100
- `message`：可读进度描述

进度估算建议：
- `prepare`：5%
- `feasible`：25%
- `optimize`：45%
- `route`：20%
- `finalize`：5%

长任务可观测性要求：
- 至少每 1~2 秒上报一次进度（或每轮迭代上报）
- 队列面板可见当前阶段、百分比、耗时
- 队列持久化策略为会话级，关闭浏览器后任务与进度全部丢失。

## 6. 失败与回退策略

典型失败：
- 基地尺寸过小，硬约束无可行解
- 某些关键连接无法满足端口规则
- 任务超时

回退策略：
1. 尝试降级目标（允许更差线长/拥塞）
2. 若仍失败，输出“失败蓝图诊断报告”（列出阻塞设备与边）
3. 不影响规划器主流程；任务可重试
4. “保底可行解”在失败态的对外呈现方式（是否附带部分蓝图对象）为开发阶段待定，需在实现前于 API 文档固化。

## 7. 与当前 Stage3 范围的一致性

- 入口在规划器，但模块架构独立。
- 自动布线仅只读领域模型，不依赖其他模块运行态。
- 蓝图必须满足“可放置在当前基地尺寸内”（本阶段仅尺寸边界约束）。
- 支持后台长任务与可视化进度。

## 8. 第一版落地建议（MVP）

第一版目标：
- 有可用解 > 求最优解
- 可中断、可重试、可观测

MVP 取舍：
- 先支持单基地矩形边界
- 先支持基础动作（move/rotate/swap）
- 先支持 A* + 简单拥塞惩罚
- 暂不承诺全局最优

## 9. 参数默认策略（Stage3）

1. 目标偏好：默认优先“可连接与可用”，其次“更少交叉”，再其次“更短总线长”。
2. 任务时间预算：默认标准档（30~60s）。
3. 失败策略：允许失败并输出诊断报告；若可产出保底可行解则优先返回可行解。
4. 待定项：保底可行解在 `failed` 状态下的对外契约（仅诊断 / 诊断 + 部分蓝图）由开发阶段拍板。
